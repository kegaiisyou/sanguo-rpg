## 1. 现状诊断（已通读代码确认）

### 1.1 三端内容与剧情对照

| 端 | 入口/数据文件 | 剧情设定 | UI/逻辑完整度 | 与本路线契合度 |
|----|--------------|---------|--------------|--------------|
| 网页 H5 | `index.html` | 应为颍川/华雄线（待对齐） | 单文件、迭代最多，但逻辑散落 | ⭐⭐⭐ 文字/DOM 最顺手，调试最方便 |
| 微信小游戏 | `wechat-game/js/data.js` + `game-core.js` | 颍川剿黄巾 / 斩华雄（5 图） | 最高：Canvas 主循环+战斗 | ⭐ 重 Canvas/美术，与弱化美术路线冲突 |
| 微信小程序 | `wechat-miniprogram/utils/gameData.js` | 幽州大营越狱（5 图） | 中：UI 脚手架完整，剧情冲突 | ⭐ 需开发者工具调试，文字迭代慢 |

### 1.2 三个核心矛盾

1. **剧情三方不一致**：`GAME_DESIGN.md` 与小程序讲"幽州越狱"，小游戏讲"颍川剿黄巾"。文档实际只和**小程序**对得上，反而和小游戏脱节。
2. **命名不统一**：版本手册/仓库叫"三国江湖 RPG"，设计文档/小程序叫"乱世烽火"——必须统一为 **乱世烽火**。
3. **快照泛滥**：根目录 50+ 个 `luanshi_fenghuo_v*.html` / `sanguo_*.html` / 美术预览 HTML，真假版本混杂。

### 1.3 容量与调试的现实约束（决定方向的关键）

- **微信端容量不是问题**：小游戏主包 4MB、总包 20MB，小程序主包 2MB、总包 20MB，文字体积极小完全够用。
- **但微信端调试是瓶颈**：必须用「微信开发者工具」在电脑跑，手机端靠「预览」二维码（约 7 天有效）或「真机调试」看 console；每改一段文字/数值都要重编译+重新扫码，**对大量文字、高频改文案的放置游戏是明显拖累**。
- **网页端手势问题可解**：缩放/下拉/橡皮筋可通过 `viewport` + `overscroll-behavior:none` + 容器 `100dvh; overflow:hidden` + 拦截 `touchmove` 完全禁用，能做出接近原生 App 的体验（代码见第 8 节）。

## 2. 平台选择：决策框架 + 明确推荐（v2 更新）

### 2.1 三端能力对比（按"文字"路线重评）

| 维度 | 网页 H5 `index.html` | 微信小游戏 `wechat-game` | 微信小程序 `wechat-miniprogram` |
|------|---------------------|------------------------|-------------------------------|
| 文字/DOM 排版 | ✅ 最优（文字即美术） | ❌ Canvas 重绘制 | ⚠️ 原生 text 受限 |
| 上架/分发 | 零审核、跨平台、扫码即开、可做 PWA | 微信内即点即玩 + 社交裂变 | 需审核，游戏体验偏重 |
| 调试成本 | ✅ 改完刷新即看，手机连局域网秒开 | ❌ 必开开发者工具+扫码 | ❌ 必开开发者工具+扫码 |
| 容量焦虑 | ✅ 无 | ✅ 文字极小无压力 | ✅ 文字极小无压力 |
| 最适合做 | **文字主发布端（单一数据源）** | 后期 Canvas 玩法扩展 / WebView 套壳分发 | **后期分发端**（套壳或重写文字 UI） |

### 2.2 推荐结论（v2）

> **主端 = 网页 H5 `index.html/`**（以根目录 `index.html` 为入口、另建 `web/` 承载模块化代码）；**微信小程序/小游戏作为后期分发端**，复用同一份 `shared/` 数据。

理由（基于"弱化美术、大量文字、放置玩法"）：
- 文字游戏的核心是"叙事文本 + 选项分支 + 数值成长"，DOM/CSS 渲染文字体验最好，且**零审核、手机局域网秒开、改完刷新即看**，完美契合"大量文字高频迭代"；
- 微信端容量虽无压力，但**调试必须开开发者工具 + 重编译 + 扫码**，对文案密集的放置游戏拖累明显，故延后；
- 现有 `wechat-game` 的 Canvas 战斗/地图资产与"弱化美术"路线冲突，不再作为主端；其颍川线**文字剧情**可抽取进 `shared/` 复用，战斗/地图表现层暂搁置。

### 2.3 如何自己判断（自检 4 问）

1. **游戏以文字为主还是画面为主？** 文字为主 → H5 优先（本路线）。
2. **要不要借微信流量/排行榜？** 要 → 后期把 H5 用 WebView 套壳成小游戏/小程序，数据仍走 `shared/`。
3. **要不要零审核跨平台（PC/安卓/iOS 浏览器）？** 要 → 必须 H5 主端。
4. **个人精力有限，只维护一份数据？** 是 → `shared/` 为唯一数据源，H5 直接引用，微信端后期复用。

## 3. 单一数据源与三端收敛策略（v2：文字数据层）

### 3.1 目标结构

```
shared/                      ← 唯一数据源（文字剧情 + 数值配置）
  story/
    sects.json              门派设定（如颍川义军/太平道/西凉军）
    skills.json             武学树（招式、修炼曲线、前置条件）
    events.json             随机事件文本（探索/奇遇/抉择分支）
    dialogues.json          NPC 对话与任务文本
  config/
    balance.js              数值公式：挂机收益、升级曲线、掉落率
    constants.js            全局常量（离线收益上限、tick 间隔等）
  index.js                  聚合导出（浏览器 import / 微信端 require）
web/                        ← H5 主端模块化代码（替代单文件 index.html）
  index.html               入口（禁缩放/下拉，见第 8 节）
  game.js                  文字引擎：叙事渲染 + 选项分支 + 数值系统
  storage.js               存档（localStorage + 离线时间戳）
index.html                 （保留）或重定向至 web/（由你决定）
wechat-miniprogram/        后期：WebView 套壳 H5，或重写文字 UI 引用 shared/
wechat-game/               后期：复用 shared/ 文字，战斗/地图表现层暂搁置
```

### 3.2 共享数据层应包含（文字路线）

- **叙事文本**：`sects`（门派）、`skills`（武学树）、`events`（随机事件）、`dialogues`（对话/任务）。
- **数值配置**：`balance`（挂机收益公式、升级经验曲线、战斗结算公式）、`constants`（离线收益上限、存档 tick 间隔）。
- **主角/存档结构**：`defaultSave()`（等级、修为、银两、门派、已解锁节点）、`offlineGain(save, now)` 离线收益计算。

> 说明：v1 中抽离的"地图/道具/NPC/敌人"因弱化美术与地图，已不再是共享层重点。原 `wechat-game/js/data.js` 的颍川**剧情文本**可迁移进 `shared/story/`，其地图坐标/战斗数据暂不使用。

### 3.3 三端当前差异点（对齐清单）

| 项目 | 小游戏 `data.js`（颍川线） | 小程序 `gameData.js`（幽州线） | 统一动作 |
|------|--------------------------|------------------------------|---------|
| 主线剧情 | 颍川→北邙→西凉→洛阳→嵩山 | 地牢→劳役场→兵营→武库→辕门 | **采用颍川线文本**，迁 `shared/story/`；弃幽州线 |
| 表现层 | Canvas 地图+回合战斗 | 小程序 UI 壳 | 文字路线下**暂不实现表现层**，只保留文本 |
| 数值 | HP/MP/ATK/DEF/金 | HP/MP/ATK/DEF/金（值不同） | 改为放置数值：等级/修为/银两/门派进度，统一进 `balance.js` |

## 4. 清理规范（仅删"明显垃圾"）

### 4.1 "明显垃圾"判定标准（满足任一即删）

1. **空文件**：字节数为 0 的 `.html` / `.js` / `.json`。
2. **损坏文件**：无法被浏览器/解析器打开，或 `<html>` 标签残缺、JSON 解析报错。
3. **字节级重复**：与另一文件 MD5 完全相同（内容一模一样，仅文件名不同）。

### 4.2 严禁删除（保护清单）

- `backup/` 整个目录（版本手册明令禁止删除）。
- `index.html`、`wechat-game/`、`wechat-miniprogram/`、`assets/`。
- 有代表性的迭代快照（即便旧，也是历史参考）。

### 4.3 保留原则

- 根目录 50+ 快照**默认保留**，只剔除 4.1 三类垃圾。
- 若想进一步瘦身，建议移入 `backup/snapshots/` 归档（非本次范围，属"仅删垃圾"之外的动作，需你另行决定）。

### 4.4 手动检测参考命令（PowerShell，需你自行运行）

```powershell
# 找出根目录所有 0 字节文件
Get-ChildItem *.html | Where-Object { $_.Length -eq 0 }

# 找出根目录 HTML 的 MD5 重复组（字节级重复）
Get-ChildItem *.html | Get-FileHash -Algorithm MD5 | Group-Object Hash | Where-Object { $_.Count -gt 1 }
```

> 以上命令只**列出**嫌疑文件，不会删除。确认后再手动删除。

## 5. 统一世界观设定草案（重新设计 · 文字版）

> 目标：一份设定同时对齐文档与三端代码，以**文字叙事 + 数值成长**呈现，弱化美术/地图，复用颍川线背景降低重写成本。

### 5.1 命名与类型统一

- 游戏名：**乱世烽火**（废弃"三国江湖 RPG"旧称，同步改版本手册/仓库描述）。
- 类型：**东汉末年 · 文字 RPG**（弱化美术与场景地图，文字即美术；核心为叙事文本、选项分支、数值成长与离线挂机）。

### 5.2 时代线与主角

- **时代**：中平年间（黄巾之乱 → 讨董卓 → 群雄割据），作为背景舞台而非强地图约束。
- **主角（原创）**：颍川草莽「你」——未被史书记载的义军头目，因乡里遭黄巾劫掠而起事，逐步累积声望与武学。
- **核心循环**：文字事件/抉择 → 数值成长（修为/银两/门派进度）→ 解锁新门派/武学/事件分支。

### 5.3 内容大纲（文字事件驱动，弱化地图）

| 阶段 | 标题 | 文字内容来源 | 状态 |
|------|------|------------|------|
| 序章 | 颍川起兵 | 复用颍川线背景文本（起义、剿黄巾、斩华雄叙事） | 📝 改写为文字事件 |
| 第一阶段 | 讨董联盟 | 关东联军、盟会抉择事件 | 📝 规划 |
| 第二阶段 | 武学成长 | 门派/武学树（拜师、修炼、突破事件） | 📝 规划 |
| 第三阶段 | 群雄割据 | 势力选择分支、奇遇事件 | 📝 规划 |

### 5.4 关键文本资产与阵营（承接现有）

- **门派（sects）**：颍川义军（玩家所属）、太平道（黄巾）、西凉军（董卓）——以文本描述门风/武学倾向。
- **关键 NPC（dialogues）**：县尉韩伯（序章引子）、医者华老、方士左慈、水镜先生、华雄（序章叙事 BOSS）。
- **武学树（skills）**：基础拳脚 → 流派招式 → 绝学，配修炼曲线与前置条件。

### 5.5 废弃设定

- **幽州大营越狱**线（地牢/劳役场/兵营/武库/辕门、幽州督军 BOSS）整体废弃；Canvas 地图/回合战斗表现层在文字路线下**暂不使用**，仅保留颍川线**文本**进入 `shared/story/`。

## 8. 附录：网页端禁缩放/下拉代码（直接可用）

`web/index.html` 的 `<head>` 与样式：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover">
```

```css
html, body { margin:0; padding:0; height:100dvh; overflow:hidden; overscroll-behavior:none; }
#game { height:100dvh; overflow:hidden; touch-action:none; }
```

```js
// 禁下拉刷新/橡皮筋/双指缩放
document.addEventListener('touchmove', e => e.preventDefault(), { passive:false });
```

> 文字游戏本不需缩放，禁用后体验更接近原生 App；`100dvh` 适配移动端地址栏伸缩。
